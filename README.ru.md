# TN Garmin MCP

[English](README.md) | **Русский**

MCP-сервер для тренировочного анализа поверх Garmin Connect: не «ещё один дашборд
с VO2max», а **обогащённый профильный доступ к твоим данным прямо в чате Claude** —
формы пульса/темпа, декуплинг, биомеханика по скорости, лактатные метки с привязкой
к секунде замера, кросс-периодные агрегаты. Методологически отвергает
device-оценки (VO2max/training-effect/пороги «из коробки») — только измеримые
факты и производные от них, без скрытых моделей.

Два слоя, оба через MCP:

- **`garmin-tn` (профильный, рекомендуемый)** — обогащённый анализ + твои
  лактатные метки и заметки. **Один коннектор = один Garmin-аккаунт/профиль.**
  Несколько людей/аккаунтов — несколько коннекторов, данные не пересекаются.
- **`garmin-raw` (сырьевой, опционален)** — необработанные пульс/каденс/мощность/
  шаг/высота по кругам и посекундно, комментарий-лактат. Один аккаунт на сервер,
  без профилей. Нужен, если хочешь работать с сырьём напрямую или выгружать JSON
  для стороннего анализа.

Если не уверен, что нужно — ставь `garmin-tn`, это основной путь.

## Требования

Python 3.10+ и [`uv`](https://astral.sh/uv). Claude Desktop для MCP-доступа
(коннекторы работают через локальный stdio — сервер и хостинг не нужны).

## Установка с нуля

```bash
git clone https://github.com/asilenin/Garmin-TN-MCP.git
cd Garmin-TN-MCP
uv sync
```

### 1. Авторизация в Garmin (один раз на аккаунт)

```bash
uv run garmin-raw-auth
```

Email, пароль, MFA-код. Токены лягут в `~/.garminconnect` — это **аккаунт по
умолчанию** (см. «Несколько профилей» ниже, если у тебя больше одного Garmin-
аккаунта, напр. свой + партнёра).

### 2. Первичное наполнение кэша

Профильный слой хранит собственную локальную базу (`~/.garmin-tn/profiles/<slug>/
cache.db`) — MCP-тулы читают из неё, в сеть не ходят (§«Как это устроено» ниже).
Наполни её один раз перед первым использованием:

```bash
uv run python garmin_raw/sync.py <slug>              # каталог тренировок (все даты)
uv run python garmin_raw/sync.py <slug> enrich 50     # обогащение + потоки, свежие первыми
uv run python garmin_raw/sync.py <slug> fetch-aux     # круги + комментарии (лактат)
```

`<slug>` — короткое имя профиля (`anton`, `mila`, латиницей/цифрами). Если тренировок
за годы много, `enrich` можно гонять пачками (`enrich 100`, потом ещё раз) — он
пропускает уже обогащённое. **На большом архиве первичное обогащение может занять
больше часа** — это ожидаемо (Garmin отдаёт посекундные потоки без пакетного API).

Проверить, что накопилось:

```bash
uv run python garmin_raw/sync.py <slug> status
```

### 3. Подключить к Claude Desktop

```bash
uv run garmin-tn-install <slug>
```

Дописывает коннектор `garmin-tn-<slug>` в `claude_desktop_config.json`, не трогая
остальное (бэкап перед записью). Путь к репозиторию определяется автоматически;
можно явно: `uv run garmin-tn-install <slug> /полный/путь/к/репо`.

Перезапусти Claude Desktop (Cmd+Q на macOS — не просто закрыть окно) — появятся
8 тулов `garmin_*`.

## Несколько профилей (разные Garmin-аккаунты)

Пример: у тебя основной аккаунт, у партнёра — свой.

```bash
# для аккаунта, авторизованного через garmin-raw-auth выше — используй его токены:
uv run garmin-tn-install anton --tokenstore ~/.garminconnect

# для второго аккаунта — сначала отдельная авторизация в его собственную папку:
mkdir -p ~/.garmin-tn/profiles/mila/tokens
GARMIN_TOKENSTORE=~/.garmin-tn/profiles/mila/tokens uv run garmin-raw-auth
uv run python garmin_raw/sync.py mila
uv run python garmin_raw/sync.py mila enrich 50
uv run python garmin_raw/sync.py mila fetch-aux
uv run garmin-tn-install mila
```

`--tokenstore` нужен, **только если** у профиля нет собственной папки токенов —
это явный флаг, не скрытый дефолт: install предупредит при установке, если токенов
нигде не нашлось. Профиль со своими токенами (`~/.garmin-tn/profiles/<slug>/tokens/`)
их не потеряет и не спутает с чужим аккаунтом.

После установки нескольких профилей в Claude одновременно доступны все —
`garmin_compact`, `garmin_add_lactate` и т.д. работают в контексте того
коннектора, который вызван; профили никогда не пересекаются (в имени тула
профиля нет намеренно — он определяется тем, какой коннектор ты используешь,
не тем, что напишешь в чате).

## Обновление кода

```bash
cd Garmin-TN-MCP
git pull
uv sync
```

Конфиг Claude трогать не нужно — коннектор хранит только путь к репозиторию,
новый код подхватится при следующем запуске Claude Desktop (перезапусти его).
Если менялась схема локальной базы — миграция применяется автоматически при
первом обращении, аддитивно (существующие данные не теряются).

## Синхронизация — держим кэш свежим

MCP-тулы **не ходят в сеть** — только читают локальный кэш (быстро, предсказуемо,
без сетевых сбоев посреди разговора). Свежие тренировки нужно **явно** затянуть:

```bash
uv run python garmin_raw/sync.py <slug>              # новые тренировки в каталог
uv run python garmin_raw/sync.py <slug> enrich 20     # обогатить свежие 20
uv run python garmin_raw/sync.py <slug> fetch-aux     # круги/комментарии недостающих
```

Гоняй это перед разговором о свежих тренировках (или периодически по крону).
`sync.py <slug> recompute` — пересчитать все метрики из уже скачанного сырья
(нужно после обновления кода, если менялась логика расчёта; сеть не требуется).

> Инициирование синка прямо из чата Claude («скачай сегодняшнюю тренировку») —
> в работе, см. backlog. Сейчас синк — отдельный шаг в терминале.

## Удаление

```bash
uv run garmin-tn-uninstall <slug>        # убрать один профильный коннектор
uv run garmin-raw-uninstall              # убрать сырьевой коннектор (если ставился)
```

Оба только правят конфиг Claude (с бэкапом) — код и данные не трогают.
Перезапусти Claude Desktop после удаления.

Полная зачистка:

```bash
rm -rf ~/.garmin-tn                # локальные базы всех профилей
rm -rf ~/.garminconnect            # токены аккаунта по умолчанию
cd .. && rm -rf Garmin-TN-MCP      # сам репозиторий
```

## Тулы (`garmin-tn`, профильный)

| Тул | Отдаёт |
|---|---|
| `garmin_status()` | что в кэше: диапазон дат, число тренировок, время последнего синка |
| `garmin_query(limit, order, filters)` | каталог тренировок по фильтру (без гистограмм) |
| `garmin_compact(activity_id)` | формы пульса/темпа, кластеры, декуплинг, биомеханика, лактат-метки |
| `garmin_full(activity_id)` | всё обогащение целиком |
| `garmin_aggregates(period_key?)` | кросс-периодные агрегаты (динамика формы) |
| `garmin_add_lactate(activity_id, mmol, …)` | внести замер лактата с привязкой к секунде (см. ниже) |
| `garmin_add_note(activity_id, text)` | внести свободную заметку к тренировке |
| `garmin_delete_mark(mark_id)` | удалить метку/заметку |

## Лактат — с привязкой к секунде замера

Есть три источника, разной ценности:

1. **Комментарий Garmin** (`LA:6.1`, можно с контекстом `LA:6.6 @rep12`) — читается
   автоматически, но это **голое число**: неизвестно, на каком пульсе/темпе он был.
2. **Часы с ConnectIQ TN Splits View** (числовое поле в потоке) — тоже читается
   автоматически, привязка к кругу есть, но требует стороннего приложения на часах.
3. **`garmin_add_lactate` — рекомендуемый способ.** Скажи в чате, когда взят замер
   (время от старта или номер круга) — тул резолвит его к конкретной секунде потока
   и возвращает пульс/темп именно в этот момент. Разница важна: «5.1 ммоль на пульсе
   129 в глубоком восстановлении» и «5.1 ммоль на пульсе 175 в работе» — разная
   физиология при одинаковом числе.

Три способа указать время (ровно один):

```
garmin_add_lactate(id, 5.5, at_elapsed_s=2190)     # 36:30 от старта (как в комментарии)
garmin_add_lactate(id, 5.5, at_ms=1782884498000)   # абсолютное время (точный путь)
garmin_add_lactate(id, 5.5, user_ref="lap14")      # конец 14-го круга по Garmin
```

Если тренировки ещё нет в кэше — сначала `sync`. Если посекундный поток ещё не
скачан — метка сохранится и привяжется автоматически при следующем `enrich`.

## Как это устроено (коротко)

MCP-тулы читают только локальную SQLite (`~/.garmin-tn/profiles/<slug>/cache.db`) —
это даёт стабильный быстрый ответ и защищает профили друг от друга: каждый
коннектор жёстко привязан к своему файлу и своим токенам через переменные
окружения в конфиге Claude (не то, что ты пишешь в чате). Наполнение кэша —
отдельная явная операция (`sync.py`), с сетью работает только она.

## Замечания

- **PR Garmin — авто-детект самого быстрого сплита**, не протокольное время; может
  быть быстрее официального. Для маркеров формы ориентируйся на протокольные забеги.
- **PII** (имя/ID владельца) вырезается из локального кэша и выдачи регистронезависимо.
- Если тулы вернули пусто/ошибку на давно синканном профиле — почти всегда протухли
  Garmin-токены: `uv run garmin-raw-auth` (или с `GARMIN_TOKENSTORE=<путь профиля>`
  для не-дефолтного аккаунта), затем `sync`.

## Сырьевой слой (`garmin-raw`, опционально)

Отдельный одноаккаунтный MCP без профилей и без обогащения — только необработанные
данные. Полезен для экспорта JSON стороннему анализу или если профильный слой не нужен.

```bash
uv run garmin-raw-install     # коннектор garmin-raw в Claude
uv run garmin-raw-export --start 2026-06-01 --end 2026-06-21   # или JSON-выгрузка
```

<summary>Тулы garmin-raw</summary>

| Тул | Отдаёт |
|---|---|
| `list_activities(start, end, sport)` | сырые сводки за период |
| `get_activity_laps(id)` | пульс/каденс/мощность/шаг/высота по кругу |
| `get_activity_streams(id)` | посекундные потоки |
| `get_activity_comment(id)` | комментарий + распарсенный `LA:x.x` |
| `get_activity_lactate(id)` | числовой лактат из TN Splits View, по кругам |
| `get_wellness(date)` | сон, HRV, RHR, стресс, Body Battery |
| `get_personal_records()` | PR по дистанциям |

</details>

## Дисклеймер про Garmin

Проект обращается к Garmin Connect **неофициальным** способом (через community-
библиотеку [`python-garminconnect`](https://github.com/cyberjunky/python-garminconnect),
которая логинится твоими учётными данными). Он **не аффилирован с Garmin, не
одобрен и не поддерживается ими**. Использование может подпадать под Условия
обслуживания Garmin; ты используешь его **на свой риск**. Гарантий нет (см.
[`LICENSE`](LICENSE)).
